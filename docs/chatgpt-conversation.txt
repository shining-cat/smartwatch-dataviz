Here’s an honest assessment + the cleanest way forward.
Why we got stuck (root causes, not symptoms)
1) We’ve been “patching artifacts,” not developing from a stable source
Each ZIP iteration is effectively a new fork of a single-file app. Small fixes (tabs, chart instances, visibility) interact in surprising ways and regress easily. A proper dev loop needs:
source files (src/…)
a build step
version control history
the ability to run/verify changes repeatedly
2) Chart libraries + tabs are a classic footgun
ECharts (and most canvas libs) will happily “render” into a container that is:
display: none, or
0×0, or
not yet laid out
Result: no errors, but nothing visible until you explicitly resize() after visibility changes. We saw this in your console trace: a parent display:none made all chart containers 0×0.
3) DOM-id globals / instance naming collisions
Browsers sometimes expose id="foo" as window.foo. That caused confusion between:
window.sleepRangeChart (HTMLElement)
“sleep range chart instance” (ECharts object)
We tried to rename to __sleepRangeChart, but because we’re not in a modular build system, it kept reappearing in other ways.
4) Tab switching is not isolating view layers
Your screenshot where Timeline’s chart appears in Sleep is a dead giveaway:
the visible chart container is still the Timeline container (#chart)
the Sleep tab doesn’t strictly own its own view surface
This is exactly the kind of thing that becomes straightforward once we structure the UI properly, but is miserable to chase in a patched single-file.
